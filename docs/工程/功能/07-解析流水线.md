# 功能 07 · 解析流水线

> UI：[figma/08-解析进度](../../../figma/08-解析进度.png)  
> 产品：[02-喂养与归类](../../产品/02-喂养与归类.md)  
> 前置：功能 04/05/06 已能产生「待解析正文」

## 1. 功能目标

把任意 source 的正文变成：切片 → Embedding → LLM 摘要与分类建议 → 状态变为「待用户确认入库」。

## 2. 流水线阶段

| 序 | 阶段 | 输入 | 输出 |
|----|------|------|------|
| 1 | 取正文 | 文件/transcript | `raw_text` |
| 2 | 清洗 | raw | 去噪、统一换行 |
| 3 | 切片 | text | `chunks[]`（约 300–800 字，重叠 50–100） |
| 4 | Embedding | chunks | `vector` 写入 pgvector |
| 5 | 摘要/分类 | 抽样或全文缩略 | `draft_summary`、`draft_tags[]`、可选 `draft_entries[]` |
| 6 | 等待确认 | — | `status=await_confirm` |

进度字段与 UI 步骤条一一对应。

## 3. 技术要点

| 项 | 做法 |
|----|------|
| 异步 | MVP：`sources` 状态机 + 后台 task；量大再上 Redis/ARQ |
| LLM | 必须 `require_llm()`；用用户 Key |
| 分类提示词 | 要求输出 JSON：`tags: string[]`（1..N）、`summary`、可选 `entries` |
| 幂等 | 同一 source 重新解析先删旧 draft chunks（未确认入库的） |
| 费用 | 超长文本先 map-reduce 摘要，避免一次塞爆上下文 |

## 4. 实现步骤

1. `POST /api/sources/{id}/start-parse`（或上传后自动开始）
2. Worker 逐步更新 `stage` / `progress` / `logs`
3. `GET /api/sources/{id}` 供前端轮询（2s）
4. 完成后写入 draft 表或 JSON 字段，供功能 08 读取
5. 失败可重试；保留错误信息

## 5. 验收

- [ ] 进度页步骤会推进，不是假进度条
- [ ] 完成后有摘要与至少一个建议标签
- [ ] 无 Key 时不能开始解析
- [ ] 中断/取消有明确状态
