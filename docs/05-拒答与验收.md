# 05 · 拒答与验收

拒答是副业的 **产品质量**，不是缺陷。写进验收，才能和「随便套个 AI」拉开差异，也减少售后扯皮。

## 1. 水质规则（给工具里的系统提示）

用客户能理解的话约束模型，大意如下：

1. 只能根据提供的知识片段回答。  
2. 片段不足或不相关 → 明确说：根据现有资料无法回答 / 超出本库范围。  
3. 禁止用通识知识编造业务承诺、价格、政策、考点。  
4. 能答时尽量指出依据来自哪段资料。

## 2. 验收题怎么设计

| 类型 | 数量 | 期望 |
|------|------|------|
| 库内明确有的 | ≥3 | 答对，最好能看出依据 |
| 库内可归纳的 | 1～2 | 谨慎归纳，不瞎扩 |
| 库外 / 未提供 | ≥2 | **必须拒答** |
| （可选）诱导越狱 | 1 | 仍拒答或仅据库内 |

客服向示例拒答题：「你们支持加密货币付款吗？」（库中未写）  
课程向示例拒答题：「帮我写一篇无关课程的申论」（超纲）

## 3. 验收记录表（每单一份）

| 题号 | 问题 | 期望 | 实际 | 通过 |
|------|------|------|------|------|
| 1 | … | 能答 | … | ✓/✗ |
| … | | 拒答 | | |

交付时附截图或表格。客户环境跑通后再算通过；不通过不收尾款/不结单。

## 4. 客户教育（避免期望落差）

开单时说清：

- 这不是全能聊天，是 **你的资料前台**。  
- 资料没写的政策/考点，系统应拒绝，这是保护你。  
- 要扩大能答范围 → **补资料**，不是改成更敢编。

## 5. 调不通时的排查顺序

1. 资料是否真的含答案  
2. 切片是否切碎/切错  
3. 检索是否召回相关段  
4. 提示词是否够硬  
5. 是否该要求客户补材料而非硬调模型
