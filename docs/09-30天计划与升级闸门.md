# 09 · 30 天计划与升级闸门

## 1. 四周日历（Docker 交付向）

| 周 | 目标 | 动作 |
|----|------|------|
| **W1** | 自己家跑通 | NAS 上 Compose 起网页知识库；能答+拒答；写出《运行说明》草稿 |
| **W2** | 可卖的包 | 固定交付目录模板；录屏；闲鱼上架「Docker 包+场景」 |
| **W3** | 第一单 | 按客户资料定制包 → 客户侧跑通 → 验收+导出 |
| **W4** | 复盘 | 客户卡在哪步就改文档/脚本；决定是否加「代部署到客户云」加价项 |

### 成功线

- 自己有可复用交付模板 + ≥1 笔付费或 ≥5 有效询盘  

### 失败信号

- 只想你托管、无人愿自备环境 → 可加清「代部署到你的云」加价试水；仍不要急着自己租云  

## 2. 时间盒

每周 6～10h：打磨 Compose/文档、获客、交付。  
**不花时间**买云做托管平台。

## 3. 升级闸门

### 仍停留在「卖 Docker 包」直到：

- 重复交付顺利、文档很少被问倒  

### 才考虑「云托管」（另开套餐）当：

1. 多笔单明确只要托管、愿加价  
2. 你算清云成本仍有利润  
3. 与 Docker 自托管套餐 **分开卖**，避免范围糊掉  

未过闸门：继续 NAS 自研 + 卖构建运行。

## 4. 禁止清单

- [ ] 默认用你家 NAS 给客户当生产  
- [ ] 未验证需求就租云做平台  
- [ ] 交空 Docker 不谈垂直与拒答  
- [ ] 包里塞真实 API Key  
- [ ] 对标扣子功能军备
